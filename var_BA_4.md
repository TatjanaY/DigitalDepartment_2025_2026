# 📈 Кейс. Бизнес-аналитика сервиса доставки еды: клиенты, рестораны и эффективность маркетинга

## 🎯 Постановка задачи

Компания развивает **сервис онлайн-доставки еды** и стремится повысить устойчивость бизнеса и эффективность маркетинговых инвестиций за счёт данных.

**Цель исследования** — определить:  
- какие **поведенческие паттерны пользователей** коррелируют с высокой ценностью;  
- как **удержание и частота заказов** меняются во времени;  
- какие **типы ресторанов и кухни** наиболее востребованы;  
- насколько **окупаются маркетинговые каналы**.

> ⚠️ **Важно**: анализ выполняется на основе **нескольких независимых датасетов**.  
> Их совместное использование носит **аналитический и концептуальный характер**, а не предполагает прямого объединения записей.

В рамках кейса необходимо провести **комплексный бизнес-анализ**, включающий расчёт ключевых метрик: **Retention, Conversion, LTV и ROI**.

**Результатом выполнения задания** должен стать **Jupyter Notebook**, содержащий код, расчёты, функции, визуализации и аналитические выводы.

---

## 📦 Используемые датасеты

В рамках кейса используются **три независимых датасета**, отражающих разные аспекты деятельности сервиса.

### 📁 Входные данные (обзор)

---

### 📄 Датасет 1 — Food Delivery Orders Dataset  
**Kaggle:** [https://www.kaggle.com/datasets/gauravmalik26/food-delivery-dataset](https://www.kaggle.com/datasets/gauravmalik26/food-delivery-dataset)  

**Тип данных:** транзакционные данные заказов  

**Основные признаки:**  
- `Order_ID` — идентификатор заказа  
- `Customer_ID` — пользователь  
- `Order_Date` — дата заказа  
- `Order_Value` — сумма заказа  
- `Restaurant_ID` — ресторан  
- `Delivery_Time` — время доставки  

**Назначение:**  
Используется как **основной источник продуктовых данных** для расчёта **выручки, retention, LTV, частоты заказов** и формирования **пользовательских профилей**.

---

### 📄 Датасет 2 — Zomato Restaurants Dataset  
**Kaggle:** [https://www.kaggle.com/datasets/shrutimehta/zomato-restaurants-data](https://www.kaggle.com/datasets/shrutimehta/zomato-restaurants-data)  

**Тип данных:** справочник ресторанов  

**Основные признаки:**  
- `Restaurant ID` — идентификатор ресторана  
- `City` — город  
- `Cuisines` — тип кухни  
- `Average Cost for two` — средний чек на двоих  
- `Rating` — рейтинг ресторана  

**Назначение:**  
Используется для **анализа ресторанных сегментов**, **сравнения кухонь и городов**, **оценки привлекательности** по рейтингу и цене.  
⚠️ **Не содержит заказов и не объединяется напрямую с транзакциями**. Используется как **самостоятельный аналитический блок**.

---

### 📄 Датасет 3 — Marketing Funnel Dataset  
**Kaggle:** [https://www.kaggle.com/datasets/olistbr/marketing-funnel-olist](https://www.kaggle.com/datasets/olistbr/marketing-funnel-olist)  

**Тип данных:** агрегированные маркетинговые данные  

**Основные признаки:**  
- `lead_id` — идентификатор лида  
- `first_contact_date` — дата привлечения  
- `origin` — канал привлечения  
- `landing_page_id` — точка входа  

**Назначение:**  
Используется для анализа **воронки привлечения**, расчёта **конверсии**, **CAC** (условного) и **ROI** (на основе модельных допущений).  
⚠️ **Не содержит прямых данных о заказах**. Связь с LTV устанавливается **аналитически, а не технически**.

## 🧩 Структура итогового Jupyter Notebook

### 1. 📊 Предварительный анализ данных  
**Цель раздела:** понять структуру, качество и ограничения данных.  

**Необходимо:**  
- вывести размеры таблиц (`shape`);  
- изучить типы данных (`info`);  
- оценить пропуски и аномалии (например, `Order_Value ≤ 0`);  
- кратко описать особенности каждого датасета.  

**В конце раздела:** краткие выводы по данным и их пригодности к анализу.

---

### 2. 🧹 Предобработка данных  
**Цель раздела:** подготовить данные к расчёту метрик и анализу.  

**Необходимо:**  
- **Для заказов**:  
  - привести `Order_Date` к `datetime`;  
  - удалить некорректные заказы (`Order_Value ≤ 0`);  
  - агрегировать данные по `Customer_ID` и временным периодам.  
- **Для ресторанов**:  
  - обработать пропуски в `Cuisines`, `Rating`;  
  - нормализовать категории кухонь.  
- **Для маркетинга**:  
  - привести `first_contact_date` к `datetime`;  
  - проверить уникальность `lead_id`;  
  - назначить **условные затраты** по каналам (например, `$1.2` для `paid`).  

**В конце раздела:** промежуточные выводы о готовности данных.

---

### 3. 🧠 Разработка функций для анализа данных  
**Цель раздела:** создать переиспользуемый аналитический инструментарий.  

**Ключевое правило:**  
> Все функции — расчёта метрик и визуализации — создаются здесь,  
> но **графики строятся и анализируются в отдельном разделе**.

**В этом разделе разрабатываются функции:**

1. **Для агрегации данных:**  
   - по пользователям;  
   - по ресторанам;  
   - по временным периодам (день / неделя / месяц).  

2. **Для фильтрации данных:**  
   - по датам;  
   - по городам (`City`);  
   - по типу кухни (`Cuisines`).  

3. **Для расчёта базовых метрик:**  
   - DAU / MAU;  
   - количество заказов;  
   - суммарная и средняя выручка.  

4. **Для формирования пользовательских профилей:**  
   - дата первой/последней покупки;  
   - количество заказов;  
   - суммарная выручка;  
   - lifetime (в днях);  
   - (опционально) RFM-сегментация.  

5. **Для расчёта продуктовых и бизнес-метрик:**  
   - Retention;  
   - Conversion;  
   - LTV;  
   - условный CAC и ROI.  

6. **Для визуализации:**  
   - line chart;  
   - bar chart;  
   - heatmap;  
   - funnel chart;  
   - scatter plot.  

**Структура раздела:**  
- краткое введение в модульность и повторное использование кода;  
- реализация функций с аннотациями;  
- перечень функций и их назначение.

---

### 4. 👤 Создание пользовательских профилей  
**Цель раздела:** сегментировать пользователей.  

**Необходимо:**  
- использовать разработанные функции;  
- рассчитать для каждого клиента:  
  - дату первой заказа;  
  - дату последней заказа;  
  - количество заказов;  
  - суммарную выручку;  
  - lifetime;  
- при желании реализовать **RFM-сегментацию**.  

**В конце раздела:** выводы по выявленным сегментам (например, «частые пользователи с высоким чеком в Дели»).

---

### 5. 📐 Расчёт продуктовых и бизнес-метрик  

#### 5.1 🔁 Расчёт удержания (Retention)  
- формирование когорт по дате первого заказа;  
- расчёт retention rate по неделям/месяцам;  
- создание retention-матрицы (когорта × период жизни).  

#### 5.2 🔄 Расчёт конверсии  
- конверсия:  
  - лид → активный клиент (совершивший заказ);  
  - по каналам привлечения (`origin`);  
  - доля повторных покупателей (на основе заказов).  

#### 5.3 💰 Расчёт LTV и ROI  
- расчёт LTV по пользователям и когортам (на основе заказов);  
- расчёт условного CAC = `затраты / лиды` по каналу;  
- расчёт ROI = (LTV – CAC) / CAC **на основе модельных допущений**.  

> **Важно:** в этом разделе используются функции, но **визуализация переносится в раздел 6**.

---

### 6. 📉 Визуализация продуктовых и бизнес-метрик  
**Цель раздела:** визуально представить и интерпретировать результаты.  

**Визуализации:**  

1. **🔁 Retention**  
   - Heatmap: X — период жизни, Y — когорта, цвет — retention %;  
   - Line chart: X — время, Y — retention %, линии — когорты.  

2. **🔄 Conversion**  
   - Funnel chart: лид → заказ;  
   - Bar chart: X — каналы, Y — conversion rate.  

3. **💰 LTV и ROI**  
   - Line chart: X — lifetime, Y — накопленный LTV по когортам;  
   - Bar chart: X — каналы, Y — ROI;  
   - Scatter plot: X — условный CAC, Y — LTV, цвет — канал.  

4. **🍽️ Рестораны**  
   - Bar chart: топ-10 кухонь по популярности;  
   - Scatter plot: X — рейтинг, Y — средний чек.  

**В конце раздела:** выводы по визуальному анализу — лучшие каналы, устойчивые сегменты, риски оттока.
